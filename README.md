## 怎么用

### 步骤 0：输入路由

用户触发 skill-mechanic 后，根据输入确定目标路径：

- if 用户说「排查skill X」（带具体名称）→ 先项目 `.claude/skills/` 再全局 `~/.claude/skills/`，查找名为 X 的目录（大小写不敏感）
  - 找到 → 继续步骤 1
  - 未找到 → 报告「未找到 skill：X，已搜索项目和全局 .claude/skills/」，终止
- if 用户说「排查」但没指定名称 → 先从当前对话上下文中提取已出现过的 skill 名称
  - 找到 1 个 → 直接作为目标，继续步骤 1
  - 找到多个 → 列出这些 skill，让用户选择
  - 上下文中无任何 skill 出现 → 询问用户「要排查哪个 skill？」，等用户回答后继续步骤 1
- if 用户给了绝对路径 → 直接用该路径，继续步骤 1

### 步骤 1-2：定位与分类

读取目标 skill 目录，判断 skill 类型（小而美 / 大型 / 升格为大型）。

### 步骤 3-4：多层排查

- 机械层（脚本）→ 混合层（Sonnet）→ 判断层（Opus self-consistency）→ 轻量验证器（Sonnet）→ 漏报抽样
- 每层独立产出 findings，最终合并去重

### 步骤 5-8：报告与优化

生成三段式报告 → 用户决策 → 执行优化循环（含回滚机制）。

## 功能列表

| 类别 | 说明 |
|------|------|
| A. 核心价值保护 | 检查核心指令是否清晰、无歧义 |
| B. 边界情况 | 检查未覆盖的边界、模糊措辞、序列结构 |
| C. 防御性设计 | 检查失败分支、前置依赖、兜底约束 |
| D. 跨 skill 依赖 | 检查依赖声明是否完整 |
| E. 规格清晰度 | 检查冗余、错误引用、MISTAKES.md 机制、内容密度 |
| F. 复杂度匹配 | 检查架构复杂度是否与问题规模匹配 |
| G. 架构 | 检查职责分离、模块边界、循环依赖 |
| H. 可维护性 | 检查重复代码、文件大小、注释 |
| R. 执行可靠性 | 检查拆分建议是否会损害执行可靠性 |
| I. 安全与隐私 | 检查输入校验、敏感信息、安全约束 |
