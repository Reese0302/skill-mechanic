# skill-mechanic

**检查并优化 Claude Skill 的 SKILL.md 文件——5 层对抗式排查，9 维度覆盖，输出可执行的三段式报告。**

> 实测 authoring-rules：66% → 97%，自动修复 8 项 + 需决策 2 项。机器侧耗时约 2 分钟（用户决策另计），机械层 0 误报，三段式报告输出。

## 为什么做这个

我做一个能跑通的 Skill容易，但是**检查**自己做的 Skill 让它变得越来越好却很麻烦且枯燥。

作为skill开发者我们应该当飞行员——设计新 Skill、推能力边界、飞向天际。但我被迫蹲在地上当机械师——逐行检查 SKILL.md 有没有歧义、边界有没有漏、失败分支有没有写。

Markdown 文件超过 100 行，我读起来就头昏脑胀了。但 SKILL.md 动辄 300、500 行。让 LLM 检查太不靠谱，它会编造问题，你分不清哪些是真的。用评分框架硬套？小 Skill 被建议拆分，大 Skill 的核心流程被打散。

所以我做了 **skill-mechanic**——一个真正的机械师。它不替你设计，不替你决策，但它能用确定性脚本 + 多层 LLM 对抗，把你做的 Skill 里的真问题找出来、假问题过滤掉，然后给你一份清单：改什么、怎么改、你选哪个。

### 设计原则

**只查规格，不查实现。** SKILL.md 是 agent 每次运行都会读的唯一入口——规格写错了，下面所有实现都是错的。resources/ 的脚本可以用 jest、vitest 测，但 SKILL.md 的规格质量没有标准测试工具。这正是 skill-mechanic 填的坑。

**脚本优先。** 70%+ 的检查用确定性脚本完成（mechanical-validator.mjs、skill-md-lint），只在需要语义判断时才调 LLM。机械层 0 token、0 误报、<1s——把这些确定性问题从 LLM 手里抢回来，剩下的才交给模型。

**自我迭代。** mechanic 不只检查你的 skill——它会给你的 skill 装上自我迭代能力。误报和用户纠正写入 MISTAKES.md，下次排查自动跳过。积累够了就回流进 SKILL.md，你的 skill 越查越准。

## 它解决什么问题

### #1：检查器自己会犯错，报告分不清真假

> 检查器说你有 10 个问题，其中 3 个是它自己编的。

**问题**：单层 LLM 审查的误报率很高。小 Skill 被建议"拆分为 3 个模块"，50 行的 SKILL.md 被硬套大框架。你分不清哪些是真问题。

**解法**：5 层对抗架构——机械层不犯错、混合层快速过滤、判断层用 self-consistency 多视角博弈（审稿人 vs 作者辩护）、验证器兜底、漏报抽样防遗漏。自动识别"小而美" Skill，只查核心指令质量，不建议拆分。

### #2：报告全是"建议"，没有结论，改了也不知道有没有改坏

> "建议你考虑一下……" ——这不是检查，这是闲聊。

**问题**：大多数 review 工具给你一堆"建议"，你得自己判断哪些是真问题。改了 SKILL.md 之后，LLM 行为可能变了也可能没变，你不知道。

**解法**：每条 finding 都有明确结论——**自动修复 / 需要你决策 / 跳过**。内置 **test-prompts 验证**，修复后自动重跑全部 prompt，失败就回滚。不是靠感觉，是靠断言。

## 快速开始

### 安装

```bash
# 方式一：通过 skills.sh 安装（推荐）
npx skills@latest add Reese0302/skill-mechanic

# 方式二：克隆到 skills 目录
git clone https://github.com/Reese0302/skill-mechanic.git ~/.claude/skills/skill-mechanic
```

### 触发

在 Claude Code 中说以下任一关键词：

```
排查 skill grill-me
检查一下这个 skill
给我的 skill 做个体检
review skill
帮我看看 skill-mechanic
排查                          ← 未指定目标，会列出所有 skill 让你选
排查 ~/.claude/skills  ← 给绝对路径也行
```

### 最简流程

```
你：检查 skill grill-me
AI：[定位目标] → [分类：小而美/大型] → [5 层排查] → [生成报告]
AI：报告如下……需要你决策的有 3 项，请回复：1A, 2不修, 3ok
你：1A, 2不修, 3ok
AI：[执行修复] → [验证] → [更新报告]
```

## Limitations

| 维度 | 它能做到 | 它做不到 |
|---|---|---|
| **检查范围** | SKILL.md 的规格质量（核心指令、边界、防御性、架构……） | resources/ 脚本、子模块代码的质量（用 jest 等标准工具） |
| **检查深度** | 5 层架构覆盖大部分问题，机械层接近 0 误报 | 0 漏报——漏报抽样是概率性的（10%-20% 章节） |
| **判断边界** | 确定性检查（字段缺失、quote 不匹配、模糊措辞） | 设计判断——"这个架构本身对不对"是你的事 |
| **语义判断** | self-consistency 降低误报率 | 消除误报——"有没有歧义"这类判断，LLM 和人可能意见不同 |
| **验证覆盖** | 自动 test-prompts 覆盖输入→输出断言 | 覆盖所有执行路径——你自己设计的用例永远更有针对性 |

> 它帮你省 80% 的 review 时间，但剩下 20% 的设计判断还是你的。

## 真实案例

### 案例 1：authoring-rules（健康度 66% → 97%）

**排查前**：description 是叙事型、触发条件模糊、前 200 字无五要素、4 阶段 14 条散文规则、无 DO NOT、无失败分支。

**排查后**：

| 维度 | Before | After |
|------|--------|-------|
| description | "大型 Skill 的结构规范化规约" | 三段式：做什么 + 触发条件 + 输出格式 |
| 触发条件 | "规范化这个 skill""整理 skill 结构" | "写 skill""创建 skill""新建 SKILL.md" |
| 前 200 字 | 无明确五要素 | 问答：做什么/输入/输出/依赖/失败处理 |
| 步骤结构 | 4 阶段 14 条散文规则 | 12 步编号流程，每步有 if-then |
| DO NOT | 无 | 5 条 MUST NOT/NEVER |
| 失败分支 | 无 | 每个关键步骤有 if-then |
| 可视化标记 | 无 | 🔴 CHECKPOINT + 🛑 STOP |
| MISTAKES.md | 无 | 五段式模板 + 回流机制 |

**具体修复**：Step 2 新增「翻译/写作类」类型 · Step 5 设计模式表新增兜底行 + if-then · Step 6 锚点改为「紧接在依赖声明之后」 · Step 9 回流动作明确定义（N=3） · Steps 4-7 新增 if-then 失败分支 · 边界情况新增「改进/扩展 + 多 skill 场景」

### 案例 2：spec-guidelines（新建 → 健康度 84%→88%）

排查发现 5 个问题，全部自动修复或列入决策项：

| 排查发现 | 修复 |
|----------|------|
| 路径判定无明确规则 | 新增「模块判定规则」算法 |
| override 机制缺失 | 新增「用户指令冲突处理」章节 |
| 噪音扫描缺白名单扩展 | Step 3.1 新增第四条扩展 |
| 子 agent 卡点无「已知不修」格式 | 新增 [卡点编号] [修复/已知不修:理由] |
| 回流阈值硬编码 N=3 | 改为用户指定或默认值 3 |

## 工作原理

### 5 层对抗式排查（点击展开）

<details>
<summary><b>展开查看完整架构图</b></summary>

```
┌─────────────────────────────────────────────────────┐
│                    用户输入                           │
│              "检查 skill grill-me"                    │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│  Layer 0：机械层（Node.js 脚本，0 token）              │
│  · quote 行号精确匹配                                 │
│  · 字段完整性校验                                      │
│  · 重复检测                                            │
│  输出：mechanical_findings + suspect_ranges            │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│  Layer 1：混合层（Sonnet，2 组结构化 prompt）           │
│  · boundary-wording：边界、模糊措辞、序列结构           │
│  · spec-interaction：歧义、触发机制、内容密度           │
│  后置校验：quote 精确匹配 → 不通过直接 drop             │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│  Layer 2：判断层（Opus，self-consistency 对抗）         │
│  · Pass 1：审稿人视角——挑剔地找问题                    │
│  · Pass 2：作者辩护——逐条反驳 + 四态裁决               │
│    (keep / soften / escalate / drop)                  │
│  · V2 drop 反驳轮：防止误 drop                        │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│  Layer 3：轻量验证器（Sonnet）                         │
│  · 触发条件：判断层产出 / confidence<0.75 / high 级别  │
│  · 裁决：keep / revise / drop                         │
│  · 最小代价原则：drop 需反证 quote                     │
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│  Layer 4：漏报抽样（10%-20% 章节随机抽检）              │
│  · 排除已产出 findings 的章节                          │
│  · 聚焦高风险类别：A（核心指令）、F（复杂度）、G（架构）│
└──────────────────────┬───────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────┐
│  合并 → 去重 → skip 记录过滤 → 结构健康度评分 → 报告   │
└──────────────────────────────────────────────────────┘
```

**四态裁决**：判断层的 Pass 2 不是简单同意/不同意，而是 keep（批评成立）/ soften（方向对但需收窄）/ escalate（比说的更严重）/ drop（判断错误）。保留中间态，避免二态丢失信息。

drop 有门槛：A/F/G 类或 severity=high 的 drop **必须提供反证 quote**。drop 还有反驳轮：专门挑战 Pass 2 的 drop 决定，防止单轮判断过度激进。

**模型分层策略**：

| 层 | 模型 | 理由 |
|---|---|---|
| 机械层 | 脚本（0 token） | 确定性检查，不涉及 LLM |
| 混合层 | Sonnet | 结构化二元判定，输入已压缩，与 Opus 差距 < 2% |
| 判断层 | Opus | 唯一需要跨段语义推理 + self-consistency 多视角博弈的层 |
| 验证器 | Sonnet | 输入 ~1k，决策空间 3 项，上游已过 Opus 过滤 |

**硬约束**：复杂任务用 Opus，其余用 Sonnet，不用 Haiku。

</details>

一句话总结：**机械层脚本兜底 + Opus 双视角对抗 + Sonnet 验证器，误报率压到个位数。**

## 9 维度排查

### A. 核心价值保护

> 原则来源：Agent 视角设计 + 决策树 > 散文

检查核心指令是否清晰、无歧义。

- 核心指令 = DO NOT 列表、🛑 STOP 指令、输出格式定义、触发条件
- if 指令有 ≥2 种合理解读 → 需决策
- if 关键指令被大段叙述淹没 → 需决策

### B. 边界情况

> 原则来源：Spec Coding + 渐进式披露

检查未覆盖的边界、模糊措辞、序列结构。

- if 用户可能输入但未处理的情况 → 自动修复
- if "应该""建议""最好"等模糊措辞 → 自动修复：改为 if-then
- B.7 序列结构：检查步骤编号和过渡词
- B.8 动作指令明确性（仅大型）：检查是否用明确动作词

### C. 防御性设计

> 原则来源：笨内核兜底 + Harness 安全区

检查失败分支、前置依赖、兜底约束。

- if 关键步骤缺失败分支 → 自动修复
- if 存在未声明的前置依赖 → 自动修复
- if 需改措辞才能加防御 → 需决策

### D. 跨 Skill 依赖

> 原则来源：Skill 原子化 + 接口契约

检查依赖声明是否完整。

- if 引用了其他 Skill 但未声明 → 自动修复
- if 依赖路径可能不正确 → 需决策

### E. 规格清晰度（仅大型）

> 原则来源：Spec Coding + 自我迭代

检查冗余、错误引用、MISTAKES.md 机制、内容密度。

- E.7 MISTAKES.md 检查点：存在性、格式、触发机制、回流机制
- E.13 内容密度：章节长度均衡性（>3 倍均值且 >1500 字 → 需决策）

### F. 复杂度匹配（仅大型）

> 原则来源：Skill 原子化 sweet spot + Harness 钟摆

检查架构复杂度是否与问题规模匹配。

- if 简单问题用了复杂架构 → 需决策
- if 复杂问题缺结构支撑 → 需决策

### G. 架构（仅大型）

> 原则来源：笨内核（机制与策略分离）

检查职责分离、模块边界、循环依赖。

- if 存在职责重叠 → 需决策
- if 存在循环依赖 → 需决策

### H. 可维护性（仅大型）

> 原则来源：sweet spot + L2/L3 分离

检查重复代码、文件大小、注释。

- SKILL.md > 40KB → 报告，建议精简
- R 类否决拆分时 → 降级为 warning

### R. 执行可靠性（仅大型）

> 原则来源：笨内核否决权 + 渐进式披露

否决项——当 H/E/G/F 建议拆分时，R 类检查"拆分后 agent 是否需要主动读取多个文件才能正确执行"。如果是 → 阻止拆分。

很多工具优化到最后，把文件拆得七零八落，agent 执行时漏读一个文件就全盘出错。R 类是整个检查体系的**安全阀**：

- if 拆分会移动核心流程、安全边界、质量门槛 → 阻止拆分，改为精简表达
- if 拆分后主流程依赖 agent 主动读取多文件 → 需决策，标注"可能降低执行可靠性"

### I. 安全与隐私

> 原则来源：Harness 三区模型

检查输入校验、敏感信息、安全约束。

- if 未校验的外部输入 → 自动修复
- if 需新增交互步骤才能保障安全 → 需决策

## 高级特性

### Skip 记录机制

用户选择"不修复"的 finding 会持久化到 MISTAKES.md，下次排查自动跳过。不反复问你已经回答过的问题。

```
你：1不修
AI：已记录跳过，下次排查自动跳过此项。
```

### 用户主权：3 个 CHECKPOINT + 回滚

修复不是盲改——整个流程有 3 个强制暂停点，不替用户做决策：

1. **回滚初始化**：修复前创建快照（git 模式或 snapshot 模式）
2. **低风险批量**：仅新增文字的项合并为一批，一键执行
3. **高风险逐项**：修改/删除的项逐个确认
4. **语义自检**：修复后立即检查改动是否引入新歧义
5. **test-prompts 验证**：重跑全部 prompt，失败就回滚
6. **健康度监控**：分数下降时暂停，让你选择回滚/保留/终止

### 小而美 Skill 特殊规则

自动识别单文件 + ≤200 行的 Skill，降级为轻量模式：

- 只查 ABC 三类（跳过 D-I, R）
- 不评分、不要求结构完整
- 不强制要求 test-prompts.json
- 重点评每句话是否在做一件事、有没有废话

### Self-Consistency 对抗

判断层的核心机制——同一批 findings 经过两个独立视角：

1. **审稿人视角**（Pass 1）：挑剔地找问题，输出 findings
2. **作者辩护视角**（Pass 2）：逐条反驳，裁决为 keep / soften / escalate / drop
3. **Drop 反驳轮**：对 Pass 2 判定 drop 的条目再审一轮，防止误杀

**最小代价原则**：keep 的代价 = 用户多看一条非问题（低）；drop 的代价 = 真问题被掩盖（高）。不对称代价反映在判断倾向中。

### MISTAKES.md 教训回流

排查过程中如果误报或用户纠正，教训自动写入 MISTAKES.md（五段式格式）。积累 ≥N 条（默认 3）时触发回流——教训转化为 SKILL.md 的规则。

```
教训 → MISTAKES.md → 积累 ≥3 条 → 归纳为规则 → 写入 SKILL.md
```

### 结构化日志

每次排查自动生成 JSON 日志（`logs/{timestamp}.json`），包含 18 个字段：各层 findings 数、drop/soften/escalate 统计、验证器触发次数、健康度分数……支持历史追踪和趋势分析。

## FAQ

<details>
<summary><b>Q: 它会不会把我的 SKILL.md 改坏？</b></summary>

不会。三层保护：① 修复前创建快照（git 模式或 snapshot 模式），随时可回滚；② 每次修复后跑 test-prompts 验证，失败自动回滚；③ 健康度分数下降时暂停，让你选择回滚/保留/终止。你也可以在决策表里全部选"不修复"——它只报告，不动手。
</details>

<details>
<summary><b>Q: 排查结果我不同意，怎么办？</b></summary>

在决策表里选"不修复"。这条 finding 会被写入 skip 记录（MISTAKES.md），下次排查自动跳过。如果 skip 记录本身有误，手动删除对应条目即可。
</details>

<details>
<summary><b>Q: 小 Skill 会不会被过度检查？</b></summary>

不会。它自动识别"小而美"Skill（单文件 + ≤200 行），只查核心指令质量（ABC 三类），不评分、不要求结构完整、不建议拆分。一个 50 行的 Skill 不会被建议"拆分为 3 个模块"。
</details>

<details>
<summary><b>Q: 它和手动 review 比，哪个更准？</b></summary>

各有所长。手动 review 擅长设计判断（"这个架构合不合理"），skill-mechanic 擅长规格检查（"这段话有没有歧义""这个步骤缺不缺失败分支"）。手动 review 30 分钟读一个 Skill，skill-mechanic 1-2 分钟扫完。建议：先用 skill-mechanic 过一遍，再人工 review 它标记的"需要你决策的"项。
</details>
